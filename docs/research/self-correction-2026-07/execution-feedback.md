# Execution Feedback as the Correction Signal: Proven Patterns for Self-Correcting Small-Model Agent Harnesses

> Research-agent report, 2026-07-22, part of the self-correction sweep (7 dimensions + adversarial fact-check). Synthesis: [../self-correction-2026-07.md](../self-correction-2026-07.md). Raw output preserved as produced; verification appendix below notes any corrections — read it before quoting numbers.

## Top takeaways

- PROVEN: Intrinsic self-critique fails, execution feedback works — Huang et al. (ICLR 2024) showed LLMs cannot self-correct reasoning without external signals, and Olausson et al. (ICLR 2024) showed weak models cannot generate useful feedback about their own code; the fix is to feed back compiler/test/runtime results, not ask the model to critique itself.
- PROVEN at 1–8B scale: raw tracebacks + a repair loop give large gains on small models — Llama 3.1 8B gained +9.8pp (HumanEval) / +16pp (MBPP) from plain sandboxed-execution tracebacks (arXiv 2604.10508), and 1–3B models gained >4 sigma from execution-feedback self-refinement while text-only refinement gained nothing (arXiv 2604.21950).
- PROVEN: cap repair at 2 rounds and stop the moment tests pass — 2 rounds capture 76–95% of achievable gains; without early stopping, refiners break already-correct code 43–62% of the time at 1–3B scale. Budget default: maxRepairs=2, stop-on-green.
- PROVEN: route by error type — syntax/name/validation errors are highly fixable by small models (SyntaxError ~100%, NameError ~50–77% repair rates) while assertion/logic failures repair only ~4–45%; a harness should auto-retry the first class and escalate (to user, or a bigger model) on the second.
- PROVEN (SWE-agent, NeurIPS 2024): guardrails and error-message design are worth benchmark points — a lint-check-and-revert on edits was worth +3.0 SWE-bench points; 51.7% of trajectories contain failed edits and edit-success probability collapses from 90.5% to 57.2% after one failure, so errors must actively steer (SWE-agent literally injects 'DO NOT re-run the same failed edit command').
- Error messages for models should be short, structured, and actionable: state what failed, where, and the suggested next action — Anthropic's production guidance says replace opaque codes/tracebacks with 'specific and actionable improvements'; SWE-agent bounds all output, shows 'Your command ran successfully and did not produce any output' for empty results, and keeps only the last 5 observations in full.
- Loop detection is a shipped, convergent pattern: OpenHands halts on 4x identical action-observation / 3x action-error / 6x alternating pairs; Gemini CLI hashes (tool,args) and flags cycles of length 1–5 repeated 5x plus 50-char content-chunk 'chanting'; Kilocode pauses after 3 identical consecutive calls; StrongDM's Attractor spec injects a steering message on repeating patterns in a 10-call window.
- Budget guards + graceful degradation are table stakes: LangGraph ships recursion_limit=25 -> GraphRecursionError; OpenAI Agents SDK ships max_turns -> MaxTurnsExceeded; smolagents' best-in-class move is forcing a final answer synthesized from history when max_steps is hit instead of dying with an exception.
- The zod-validation re-ask loop is the proven structured-output repair: Instructor (13.6k stars, 3M monthly downloads) feeds validation errors back for up to N retries; Vercel AI SDK ships experimental_repairToolCall for NoSuchToolError/InvalidToolInput. Mithril already has zod at the tool boundary — feeding zodError.issues (paths + expected types) back as the tool result is the single highest-leverage self-correction feature for tiny models.
- Anecdotal but consistent: injected reflection messages on loop detection ('you've called X with the same args N times — explain what you need, choose a different strategy') roughly halve wasted tokens in practitioner reports; treat as a cheap default, not a guarantee.

# Execution Feedback as the Correction Signal

## 1. The core result: extrinsic beats intrinsic (PROVEN, replicated)

The foundational negative result is [Large Language Models Cannot Self-Correct Reasoning Yet](https://arxiv.org/abs/2310.01798) (Huang et al., ICLR 2024, arXiv 2310.01798): without external feedback, asking a model to review its own answer does not improve it and often degrades it. The complementary result for code is [Is Self-Repair a Silver Bullet for Code Generation?](https://arxiv.org/abs/2306.09896) (Olausson et al., ICLR 2024, arXiv 2306.09896): self-repair gains are modest once you account for the extra calls, models "are held back by their inability to reliably produce accurate and useful feedback on why the code is wrong," and swapping in stronger feedback (GPT-4 critiquing CodeLlama-13B; human feedback boosting repaired-program count 1.58x) fixes it. Conclusion for a harness: **never ask a small model to critique itself; give it ground-truth signals — validation errors, tracebacks, test results.**

The positive results all use execution as the signal. [Self-Debugging](https://arxiv.org/abs/2304.05128) (Chen et al., ICLR 2024, arXiv 2304.05128) improves accuracy up to +12% when unit-test execution results are fed back (only +2–3% from explanation alone). [Reflexion](https://arxiv.org/abs/2303.11366) (Shinn et al., NeurIPS 2023, arXiv 2303.11366) hit 91% pass@1 on HumanEval with GPT-4 — but crucially by reflecting on *execution feedback from self-generated unit tests*, not free-form self-critique.

## 2. Evidence specifically for 0.5B–8B models (PROVEN, 2024–2026)

- **[Feedback Over Form](https://arxiv.org/html/2604.21950)** (arXiv 2604.21950, 2026) tested Gemma3-1B, Qwen2.5-1.5B, Llama3.2-3B locally under Ollama: execution-feedback self-refinement improved HumanEval/MBPP by >4 sigma (Llama3.2-3B: 46.7%→57.3% HumanEval), while text-only refinement produced no gains. Pipeline topology mattered far less than having the execution signal. Two hard warnings: **without early stopping, every extra refinement round was net-negative** (43–62% regression on already-passing code), and refiner capability matters more than generator (a 1.5B generator + 3B refiner matched a 3B doing both).
- **[How Many Tries Does It Take?](https://arxiv.org/html/2604.10508)** (arXiv 2604.10508, 2026): with nothing fancier than raw error type + traceback from sandboxed execution, Llama 3.1 8B gained +9.8pp HumanEval / +16.0pp MBPP — "modern instruction-tuned models succeed with prompting alone, even at 8B scale." **Two repair rounds capture 76–95% of achievable gains.** Repair rates split sharply by error class: NameError ~77%, AssertionError ~45%.
- **[CYCLE](https://arxiv.org/abs/2403.18746)** (arXiv 2403.18746, OOPSLA 2024): fine-tuning 350M–3B models to consume execution feedback boosts self-refinement up to +63.5%; CYCLE-350M beats StarCoder-1B. (Training-side evidence that tiny models *can* use execution feedback when the format is consistent — an argument for a stable, learned-friendly error envelope.)
- **[FeedbackEval](https://arxiv.org/html/2504.06939v2)** (arXiv 2504.06939, 2025) ranked feedback types across five models: mixed 63.6% > expert 62.9% > **test feedback 57.9%** > minimal 53.1% > compiler 49.2% > unstructured LLM suggestions 48.8% (worse than "code is wrong, fix it"). Improvements plateau after 2–3 iterations. Removing docstrings/problem restatement hurt more than removing environmental context — keep the *intent* in the repair prompt.

Both 2026 papers agree on the mechanism: tracebacks pinpoint syntax/name/runtime errors (SyntaxError ~100% fixable, NameError 50–67%) but only report *symptoms* of logic errors (AssertionError 4–24% at 1–3B). **Pattern: auto-retry the mechanical class, escalate the logical class.**

## 3. Error-message design: what the best scaffolds ship (PROVEN ablations + production docs)

[SWE-agent](https://arxiv.org/abs/2405.15793) (NeurIPS 2024, arXiv 2405.15793; repo ~19.9k stars) is the canonical ACI study. Its lint-check-with-revert guardrail on edits was worth **+3.0 SWE-bench points** (score drops to 15.0 without it). Failed edits are the dominant failure: 51.7% of GPT-4 trajectories contain at least one, and while any edit attempt eventually succeeds 90.5% of the time, that drops to **57.2% after a single failure** — errors compound, so preventing/steering the first failure is disproportionately valuable. Concrete formatting choices ([deep-dive](https://dev.to/truongpx396/swe-agent-deep-dive-build-your-own-guide-ade)): every output is bounded; empty output is made explicit ("Your command ran successfully and did not produce any output" — silence confuses models); search results are capped (refuse + ask to narrow rather than truncate silently); only the last 5 observations are kept in full; and the failed-edit message injects imperative steering: "DO NOT re-run the same failed edit command."

Anthropic's production guidance ([Writing effective tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents), 2025) says errors should "clearly communicate specific and actionable improvements, rather than opaque error codes or tracebacks," and truncations should carry instructions for what to do instead. Their [SWE-bench scaffold post](https://www.anthropic.com/engineering/swe-bench-sonnet) shows error-*proofing by design*: absolute-path-only params, and edit-by-string-replacement that errors with a retryable message unless exactly one match exists. So: **short, structured, one suggested next action; design tool signatures so the common mistake is impossible.** (No paper directly A/B-tests error verbosity for small models; the closest evidence is FeedbackEval's structured-beats-unstructured ranking plus the 2026 papers' raw-traceback sufficiency — label "short + structured + actionable" as strongly-supported best practice, not a measured law.)

## 4. Loop detection: shipped heuristics (production, convergent)

- **[OpenHands StuckDetector](https://docs.openhands.dev/sdk/guides/agent-stuck-detector)** (~81.7k stars, on by default): halts on (a) same action→same observation 4+ times, (b) same action→error 3+ times, (c) 3+ agent messages with no user input ("monologue"), (d) two action-observation pairs alternating 6+ cycles, (e) repeated context-window errors. Comparison is semantic — tool name + content, ignoring IDs/timestamps.
- **[Gemini CLI loopDetectionService.ts](https://github.com/google-gemini/gemini-cli/blob/main/packages/core/src/services/loopDetectionService.ts)** (~106k stars): hashes (tool, args); flags cycles of length 1–5 repeated 5x over the last 25 calls. Adds *content chanting* detection: 50-char sliding-window chunks, loop if the same chunk hash appears 10x within ≤5x chunk-size average spacing (skipping code blocks/markdown structure to avoid false positives). After 30 turns it adds an LLM-based loop judge (cheap model first, double-check at confidence >0.9, re-check interval 5–15 turns).\n- **Kilocode/Hermes** "doom loop": pause for user confirmation after [3 identical consecutive (name, args) calls](https://github.com/NousResearch/hermes-agent/issues/512). **[StrongDM Attractor spec](https://github.com/strongdm/attractor/blob/main/coding-agent-loop-spec.md)**: window of 10 calls, repeating patterns of length 1–3, then inject "Loop detected… Try a different approach." Practitioner reports (ANECDOTAL) claim reflection-injection halves wasted tokens.\n- SWE-agent's per-node false-positive doctrine applies: its guardrails (cost cap, syntax check, lint+revert) only fire when something is *definitely* wrong — don't ship a detector whose false-positive rate you haven't measured (OpenHands' issue tracker is full of long-running-command false positives).

## 5. Budget guards and graceful degradation (production)

[LangGraph](https://docs.langchain.com/oss/python/langgraph/errors/GRAPH_RECURSION_LIMIT) (~37.8k stars) ships `recursion_limit` (default 25) → typed `GraphRecursionError`; OpenAI Agents SDK ships `max_turns` → `MaxTurnsExceeded`. The best degradation UX is [smolagents](https://github.com/huggingface/smolagents/blob/main/docs/source/en/guided_tour.md) (~28.5k stars): on hitting `max_steps` it makes one final `provide_final_answer` call that synthesizes a best-effort answer *from the run history* rather than throwing. Attractor's spec adds numbers worth copying: default command timeout 10s (cap 600s), per-tool output truncation caps (shell 30k chars, read 50k), context-window warning at 80%, and a defined graceful shutdown sequence. Tool failures never throw — they return `is_error: true` results the model can react to.

## 6. Structured-output repair (production, directly maps to zod tools)

[Instructor](https://python.useinstructor.com/) (13.6k stars, ~3M monthly downloads) is the proven pattern: on schema-validation failure, re-ask with the validation error appended, up to N retries. [Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling) (25.7k stars) ships `experimental_repairToolCall` — a hook receiving the failed call + `NoSuchToolError`/`InvalidToolInputError` that may return a fixed call (e.g., via a structured-output model) or null. For local models, constrained decoding is the stronger guarantee — [dottxt's rebuttal](https://blog.dottxt.ai/say-what-you-mean.html) to [Let Me Speak Freely](https://arxiv.org/abs/2408.02442) found structured generation helps across the board when prompted correctly (the "hurts reasoning" claim is CONTESTED) — consistent with Mithril's roadmap note that grammar decoding is the eventual fix and repair-loops are the shipped-today fix.

## 7. Concrete patterns for Mithril's middleware/loop

1. **Typed tool-error envelope, never throw**: tool failures become `{ ok: false, errorType, message, suggestedFix? }` results (Attractor's `is_error`), rendered short + actionable (Anthropic), with explicit empty-output text (SWE-agent).\n2. **Zod re-ask middleware** (default-on): on argument validation failure, return the flattened `zodError.issues` (path, expected, received) as the tool result with "fix the arguments and call again"; cap at 2 retries; expose a `repairToolCall`-style hook for BYO escalation. Highest leverage for 0.5B–4B playground models.\n3. **Repair-loop budget defaults**: `maxRepairs: 2`, stop-on-green (run verification *first*, never refine passing output), error-type routing — auto-retry syntax/name/validation, emit an escalation event for assertion/logic failures.\n4. **Loop-detector consumer** over the `MithrilEvent` stream: hash (tool, argsJSON); thresholds ~3 identical consecutive calls or patterns length 1–3 in a 10-call window; action = inject steering message once, then halt with a typed `loop_detected` event. Semantic comparison (ignore IDs/timestamps).\n5. **Budget guards as first-class run options** (`maxTurns`, `maxTokens`, `timeoutMs`) emitting typed events, with smolagents-style forced final synthesis instead of a bare throw; sandbox execution stays in the Web Worker in-browser (QuickJS-WASM/[Pyodide-style](https://github.com/langchain-ai/langchain-sandbox) isolation server-side).
