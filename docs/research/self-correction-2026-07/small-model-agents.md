# Small-Model Agents (0.5B–8B): Proven Reliability Techniques for a TypeScript Harness

> Research-agent report, 2026-07-22, part of the self-correction sweep (7 dimensions + adversarial fact-check). Synthesis: [../self-correction-2026-07.md](../self-correction-2026-07.md). Raw output preserved as produced; verification appendix below notes any corrections — read it before quoting numbers.

## Top takeaways

- Model choice is the cheapest reliability lever: function-calling fine-tunes (xLAM-2-1b/3b, Hammer-1.5B/4B/7B) beat generalist models 5-30x their size on BFCL; xLAM-2-3b scores 65.74% overall on BFCL-v3 settings vs 62.04% for Qwen3-4B prompt-mode — but multi-turn collapses below ~3B (xLAM-2-1b: 8.38%, Qwen3-0.6B: 1.38%), so keep small-model agent loops single-turn-shaped with externalized state.
- Few-shot exemplars are the single most proven prompt-side lift for tiny models: on a 1.3B model, 3 in-context examples took JSON parsability from 7.34% to 89.38% and task accuracy from 1.11% to 55.65% (arXiv 2504.19277); zero-shot tool calling at ≤4B is effectively broken (most models: 0% parsable JSON).
- Tool-count reduction is proven and dramatic: accuracy falls off a cliff with tool count (84-95% at ~50 tools → 41-83% at 200 → near-random at 740), and RAG-MCP's retrieval-based tool selection tripled accuracy (13.62% → 43.13%) while halving prompt tokens — a top-k tool-RAG middleware is the highest-ROI feature for small models.
- Validation-error-driven retry (the 'reask' loop) is the production-proven self-correction pattern: PydanticAI's ModelRetry and Instructor's reask feed typed validation errors back to the model with a bounded retry budget — near-zero cost on the happy path, and it maps 1:1 onto a zod-validation middleware in Mithril's loop.
- Adopt a typed error taxonomy: ToolScan (arXiv 2411.13547) defines 7 canonical tool-use error classes (hallucinated function name, hallucinated arg name, wrong arg value, wrong arg type, insufficient calls, repeated calls, invalid format) — encoding these as MithrilEvent error codes enables targeted retry prompts and clear DX errors; PALADIN showed error-aware recovery training + retrieved recovery exemplars lifts an 8B model's recovery rate from 21.8% to 79.8%.
- Constrained/grammar decoding helps or is neutral, never harmful, when prompts are held constant (dottxt's controlled rebuttal: structured beat unstructured on all 3 tasks on Llama-3-8B); in-browser it's available today via WebLLM+XGrammar (JSON-schema and EBNF) but NOT in Transformers.js — a real gap for Mithril's current local target, worth noting on the roadmap.
- In-browser reality: 0.5B-3B at Q4 quantization is the practical envelope (Transformers.js/ONNX q4/q4f16; wllama has a hard 2GB-per-file ArrayBuffer limit, mitigated by GGUF splitting; WebLLM retains ~80% native speed via WebGPU); streaming tool-call tag parsing is a known fragility point (mid-turn stops, malformed tags), so the parser must be error-tolerant and retry-aware.
- NVIDIA's SLM position paper (arXiv 2506.02153) is argued, not benchmarked — but its case-study estimates (40-70% of agent LLM calls replaceable by SLMs in MetaGPT/Open Operator/Cradle) and its 'agentic subtasks are narrow, repetitive, non-conversational' framing justify Mithril's design bet; treat the decomposition-into-micro-steps advice as well-reasoned but anecdotal.
- Self-consistency voting is proven for reasoning benchmarks (up to +27.6%) but has thin direct evidence for tool calling; it costs N× inference, so ship it (if at all) as an opt-in strategy for tool selection only, not a default.
- Small-model failure modes are now well-documented and predictable: format non-compliance, uncontrolled generation past the answer, hallucinated tool/argument names, wrong arg types, premature stopping, and repeated identical calls — a harness that detects each class and answers with a targeted, structured correction prompt covers the overwhelming majority of observed failures.

# Small-Model Agents (0.5B–8B): What's Proven, What's Hype, and How It Maps to a TS Harness

## 1. The case for small-model agents — and its limits

NVIDIA's position paper [Small Language Models are the Future of Agentic AI](https://arxiv.org/abs/2506.02153) (arXiv 2506.02153, Jun 2025) argues SLMs are "sufficiently powerful, inherently more suitable, and necessarily more economical" for most agent invocations because agentic subtasks are "repetitive, scoped, and non-conversational." Its evidence is a capability survey — Phi-2 (2.7B) matching 30B models on reasoning/code at ~15× speed, SmolLM2 (135M–1.7B) matching 14B contemporaries on tool calling, Hymba-1.5B, Nemotron-H, xLAM-2-8B "surpassing GPT-4o and Claude 3.5 on tool calling" — plus case-study **estimates** that 60% (MetaGPT), 40% (Open Operator), and 70% (Cradle) of LLM calls could be handled by specialized SLMs, and a 6-step LLM→SLM conversion algorithm (log calls → curate → cluster tasks → select SLM → LoRA fine-tune → iterate). **Status: a position paper — the framing and conversion recipe are argued, not benchmarked.** NVIDIA maintains a [correspondence page](https://research.nvidia.com/labs/lpr/slm-agents/correspondence.html) collecting community critiques.

## 2. What the benchmarks actually say about small models (PROVEN)

The [Berkeley Function-Calling Leaderboard](https://gorilla.cs.berkeley.edu/leaderboard.html) (from the [gorilla repo](https://github.com/ShishirPatil/gorilla), ~13k stars, active; V4 announced 2025-07-17) is the reference benchmark; V4 adds web search, memory, **format sensitivity** (scores measurably move under prompt-format variation — see the [format-sensitivity blog](https://gorilla.cs.berkeley.edu/blogs/17_bfcl_v4_prompt_variation.html)) and hallucination measurement (relevance/irrelevance detection).

Verified small-model numbers (BFCL, via the TinyLLM paper): **xLAM-2-3b-fc-r 65.74% overall** (88.22% non-live AST, 55.62% multi-turn); Qwen3-4B (prompt) 62.04% overall but **35.25% multi-turn**; Qwen3-1.7B 55.49% (16.88% multi-turn); xLAM-2-1b-fc-r 53.97% (**8.38% multi-turn**); Qwen3-0.6B 45.76% (**1.38% multi-turn**). A [BFCL-V4 snapshot](https://llm-stats.com/benchmarks/bfcl-v4) shows the same shape in newer families (Qwen3.5-4B 0.503 vs 9B 0.661). **The replicated pattern: single-turn tool calling survives down to ~1B; multi-turn state-keeping collapses first.** Design consequence for Mithril: keep small-model loops single-turn-shaped — externalize state into the harness (explicit scratchpad/memory events), don't rely on the model's conversation tracking.

**Specialized fine-tunes punch far above weight (PROVEN, replicated):**
- [xLAM-2 / APIGen-MT](https://arxiv.org/pdf/2504.03601) (Salesforce; [repo](https://github.com/SalesforceAIResearch/xLAM), ~634 stars): simulated agent-human interplay data; xLAM-2-70b beats GPT-4o on τ-bench (56.2% vs 52.9%); the 1B/3B variants ship as open weights and top same-size peers.
- [Hammer](https://arxiv.org/abs/2410.04587) (arXiv 2410.04587, ICLR-reviewed): **function masking** — replace function/param names with irrelevant tokens during training so the model attends to *descriptions*, not names — plus 7.5k irrelevance-detection examples. Hammer-1.5B: 73.04%, 4B: 76.05%, 7B: 83.92% on BFCL-v2, competitive with GPT-4-class. The transferable insight for a harness: **descriptions are the load-bearing part of a tool schema; names mislead small models.** Mithril's DX should push (and lint for) rich tool descriptions.
- 2025–26 practitioner shortlists ([KDnuggets, May 2026](https://www.kdnuggets.com/5-small-language-models-for-agentic-tool-calling)) converge on SmolLM3-3B (dual tool interfaces: XML blobs and Python-style calls — [model card](https://huggingface.co/HuggingFaceTB/SmolLM3-3B)), Qwen3-4B-Instruct-2507, Phi-3-mini, Gemma, Mistral-7B. [LFM2/LFM2.5](https://www.liquid.ai/blog/liquid-foundation-models-v2-our-second-series-of-generative-ai-models) (350M–1.2B) claims 2× CPU decode speed vs Qwen3 and explicitly positions the 350M for "tool use, data extraction, structured outputs" — **vendor-claimed**, not independently BFCL-verified.

## 3. Techniques proven to lift small-model reliability

**Few-shot exemplars — the strongest quantified prompt-side result.** [Small Models, Big Tasks](https://arxiv.org/html/2504.19277) (arXiv 2504.19277) evaluated five ≤4B SLMs: **zero-shot, most produced 0% parsable JSON** (best: DeepSeek-coder-1.3B at 7.34% parsability, 1.11% task accuracy). Three in-context examples lifted that model to **89.38% parsability / 55.65% task accuracy**; LoRA fine-tuning reached 99.44% / 85.43% and cut prompt-injection degradation from 13–16% to 1–2%. Cost: a few hundred prompt tokens per tool (prefill cost matters in-browser). **Mithril mapping:** let `asTool`/tool definitions carry optional `examples`; inject them automatically when the resolved model is below a size threshold.

**Tool-count reduction / tool RAG — proven, dramatic.** [RAG-MCP](https://arxiv.org/abs/2505.03275) (arXiv 2505.03275): semantic retrieval of top-k tools before the model sees them **tripled selection accuracy (13.62% → 43.13%) and cut prompt tokens >50%**. Independent 2025 measurements ([production-architecture synthesis](https://zylos.ai/research/2026-04-16-tool-augmented-llm-agents-production-architecture/), [tool-RAG overview](https://webscraft.org/blog/tool-rag-scho-robiti-koli-u-agenta-zabagato-instrumentiv?lang=en)) replicate the cliff: 84–95% accuracy at ~50 tools, 41–83% at 200, near-random at 740, with lost-in-the-middle position effects. **Mithril mapping:** a `toolSelect` middleware (embed descriptions once, retrieve top-k per turn) plus a dev-mode warning when tool count × model size crosses a threshold. This is the single highest-ROI reliability feature for small models.

**Validation-driven retry ("reask") — production-proven, shipped in major frameworks.** [PydanticAI's ModelRetry](https://ai.pydantic.dev/api/exceptions/) lets tool validators and output validators raise a typed error whose details are sent back to the model as a retry prompt under a `max_retries` budget; Instructor popularized the same pattern. Zero overhead on success; 1–2 extra calls on failure. **Mithril mapping:** when zod validation fails, don't surface a raw error — emit a `tool.retry` MithrilEvent and feed a structured, human-readable diff of the validation failure back into the loop, with a bounded budget and a loud terminal error after exhaustion (clear-errors DX requirement).

**Constrained/grammar decoding — proven neutral-to-positive when done right.** The claim that structure hurts ("Let Me Speak Freely") was rebutted by dottxt's controlled replication [Say What You Mean](https://blog.dottxt.ai/say-what-you-mean.html): with identical prompts on Llama-3-8B, structured beat unstructured on all three tasks (GSM8K 78% vs 77%, Last Letter 77% vs 73%, Shuffle 44% vs 41%). The 2504.19277 authors explicitly recommend constrained decoding for SLM function calling. In-browser, [WebLLM has integrated XGrammar](https://blog.mlc.ai/2024/06/13/webllm-a-high-performance-in-browser-llm-inference-engine) (JSON-schema + EBNF, with structural tags aimed at "reliable tool use with small models"); **Transformers.js has no grammar support** — Mithril's chat-template tag parsing is the only guardrail there. This is an honest roadmap item: hard structured-output guarantees on local models need either a WebLLM-class backend or native constrained decoding (already listed in Mithril's not-shipped section).

**Error-aware self-correction — proven in research, small models included.** [ToolScan](https://arxiv.org/abs/2411.13547v2) (arXiv 2411.13547) established the canonical 7-class tool-use error taxonomy: **Incorrect Function Name (hallucinated tools), Incorrect Argument Name, Incorrect Argument Value/missing required args, Incorrect Argument Type, Insufficient API Calls (premature stopping), Repeated API Calls (loops), Invalid Format** — exactly the small-model pathologies the prompt asks about, observed even in frontier models and far more frequent in small ones (2504.19277 adds "uncontrolled generation": continuing past the answer or starting an unrelated query). [PALADIN](https://arxiv.org/html/2509.25238v1) (arXiv 2509.25238) trained recovery behavior via failure injection over this taxonomy + **inference-time retrieval of 55+ recovery exemplars**: Llama-3.1-8B's recovery rate rose **21.8% → 79.8%**; Gemma-12B hit 89.68% recovery vs 23.75% vanilla. For multi-agent-level failures, [MAST](https://arxiv.org/abs/2503.13657) (arXiv 2503.13657) provides a validated 14-mode taxonomy (κ=0.88) whose headline finding — most failures are system-design, not model-capability — supports investing in harness-level orchestration. **Mithril mapping:** encode the ToolScan classes as typed error codes on MithrilEvent; per-class targeted correction prompts (e.g., hallucinated tool → "choose from this exact list"; repeated call → inject prior call+result and ask for a different action); PALADIN's retrieval trick becomes a static map of error-class → recovery snippet, no fine-tuning required.

**Self-consistency voting — proven for reasoning, thin for tool calling.** Majority-vote across samples yields up to [+27.6% on reasoning benchmarks](https://www.emergentmind.com/topics/self-consistency-in-language-models), and voting on short tool-selection outputs is cheap relative to full generations — but direct tool-calling evidence is sparse and "voting reduces variance, not bias." **Status: promising/anecdotal for this use; ship opt-in, never default (N× cost).**

**Decomposition into micro-steps.** Argued by the NVIDIA paper (specialized SLM per narrow subtask) and consistent with the multi-turn collapse data, but no controlled decomposition-vs-monolith benchmark for SLMs surfaced. **Status: well-reasoned, anecdotal.**

## 4. In-browser constraints (Transformers.js / WebLLM / wllama)

- **Practical envelope:** 0.5B–3B at Q4 is the 2026 in-browser sweet spot ([browser-LLM guides](https://pockit.tools/blog/run-llms-browser-webgpu-transformers-js-chrome-built-in-ai-guide/)); WebGPU covers ~85–90% of browser traffic (blog-claimed). [WebLLM](https://github.com/mlc-ai/web-llm) retains ~80% of native speed via precompiled WebGPU kernels ([paper](https://arxiv.org/html/2412.15803v2)) and is the only browser stack with grammar-constrained decoding today.
- **[wllama](https://github.ngxson.com/wllama/docs/)** (llama.cpp WASM): hard **2GB-per-file ArrayBuffer limit** — split GGUF via `llama-gguf-split`; Q4/Q5/Q6 recommended; runs in a Web Worker; WebGPU support has landed upstream. Relevant to Mithril's planned CPU-guaranteed backend.
- **Known fragility:** streaming tool-call tag parsing breaks on mid-turn stops and malformed tags (community reports around Qwen3-family streaming parsers). Mithril's parser should be incremental, error-tolerant, and wired into the retry loop rather than throwing. Fine-tuned dtype pinning issues (already in Mithril's CLAUDE.md) match the community experience: quantization choice is per-architecture, not universal.

## 5. Priority-ordered recommendations for Mithril

1. **Small-model mode** (auto-triggered by resolved model size): tool RAG top-k, auto-injected few-shot examples, single-turn-shaped loop with harness-held state. (All PROVEN.)
2. **Zod-reask middleware** with typed ToolScan-style error codes, per-class correction prompts, bounded retries, loud final errors. (Production-proven pattern + research-proven taxonomy.)
3. **Static recovery-exemplar map** (PALADIN-lite) injected on tool-execution failure. (Research-proven on 8B.)
4. **Recommend/bundle function-calling fine-tunes** (xLAM-2-3b, Hammer, Qwen3-4B, SmolLM3-3B) in the local-model registry, each gated through focused local-model fixtures. (PROVEN.)
5. **Roadmap honestly:** grammar-constrained decoding for local models (WebLLM/XGrammar-class) as the endgame for hard guarantees; self-consistency voting as experimental opt-in.
