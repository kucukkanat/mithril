Not every decision needs a model. A **workflow** is plain, deterministic routing over a shared
typed state — cheap branching *before* and *around* LLM calls. A step may run an agent, but the
routing itself is code you can read, so there is no hidden LLM router unless you write one.

## Define, route, finish

`defineWorkflow` takes a map of named steps and a `start`. Each step returns `goto(next, state)` to
continue or `done(state)` to stop. `run` folds from the start and returns the final `state` plus the `path` it took (highlighted). Run
it — no model involved:

<Runnable title="refunds.ts" highlight={["24-25"]} code={`import { defineWorkflow, goto, done } from "@mithril/workflows";

interface Refund {
  readonly orderId: string;
  readonly amountUsd: number;
  readonly approved?: boolean;
  readonly reason?: string;
}

const refunds = defineWorkflow<Refund>(
  {
    triage: (s) => (s.amountUsd <= 100 ? goto("autoApprove", s) : goto("review", s)),
    autoApprove: (s) => done({ ...s, approved: true, reason: "under threshold" }),
    review: (s) =>
      s.amountUsd <= 1000
        ? done({ ...s, approved: true, reason: "within policy" })
        : goto("escalate", s),
    escalate: (s) => done({ ...s, approved: false, reason: "needs manager sign-off" }),
  },
  { start: "triage", maxSteps: 16 },
);

const { state, path } = await refunds.run({ orderId: "A-1", amountUsd: 40 });
console.log("path:", path); // ["triage", "autoApprove"]
console.log("state:", state); // { …, approved: true, reason: "under threshold" }`} />

State is threaded immutably — each step receives the current state and returns the next. Nothing is
mutated in place.

## A step can call an agent

The routing is deterministic, but a step is free to `await` anything — including an agent run
(highlighted) — before it decides where to go. The *model* classifies; the *code* routes:

<Runnable title="agent-step.ts" highlight={["14-15"]} code={`import { agent } from "mithril";
import { defineWorkflow, goto, done } from "@mithril/workflows";
import { scriptedProvider, testModel, textTurn } from "@mithril/core/testkit";

interface Ticket { readonly reason: string; readonly route?: string }

const classifier = agent({
  model: testModel(scriptedProvider([textTurn("fraud")])),
  instructions: "Classify the ticket as 'fraud' or 'normal'. Reply with one word.",
});

const triage = defineWorkflow<Ticket>(
  {
    classify: async (s) => {
      const r = await classifier.run(s.reason);
      return goto(r.status === "completed" && r.output.includes("fraud") ? "escalate" : "review", s);
    },
    escalate: (s) => done({ ...s, route: "escalate" }),
    review: (s) => done({ ...s, route: "review" }),
  },
  { start: "classify" },
);

const { state, path } = await triage.run({ reason: "unauthorized charge" });
console.log(path, "→", state.route);`} />

## Termination is guaranteed

`maxSteps` (default 100) bounds the walk. Referencing an unknown step, or exceeding the budget
(a cycle), throws a `WorkflowError` rather than looping forever — failure is loud and typed.

<Aside type="tip" title="Workflow vs. agent">
Reach for a workflow when the control flow is known and deterministic (routing, validation,
fan-out). Reach for an [agent](/guides/building-agents/) when the *model* should decide the next
step. They compose — agents inside steps, workflows inside tools.
</Aside>
