An agent binds a **model**, **instructions**, and (optionally) **tools** into a loop you can `run`
or `stream`. Everything past the 90% path is one more argument — never one more import:

```ts
import { agent, tool } from "mithril";
import { anthropic } from "mithril/anthropic";

const assistant = agent({
  model: anthropic("claude-fable-5"), // self-wires the provider
  instructions: "You are a concise assistant.",
  tools: [weather],
  maxSteps: 8, // cap the model↔tool round-trips (default 16)
});

const result = await assistant.run("Weather in Istanbul?");
if (result.status === "completed") console.log(result.output);
```

The same agent runs **offline in your browser** below (a scripted `model` replaces the real handle;
`maxSteps` highlighted) — click **Open in playground**:

<Runnable title="agent.ts" highlight={[14]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

const weather = tool({
  name: "weather",
  description: "Current weather for a city.",
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ city }) => ({ city, tempC: 21 }),
});

const model = testModel(scriptedProvider([toolCallTurn("weather", { city: "Istanbul" }), textTurn("It's 21°C in Istanbul.")]));
const assistant = agent({
  model,
  instructions: "You are a concise assistant.",
  tools: [weather],
  maxSteps: 8,
});

const result = await run(assistant, "Weather in Istanbul?");
if (result.status === "completed") console.log(result.output);`} />

## Model handles vs. registry strings

A provider handle self-wires — `anthropic("claude-fable-5")`, `openai("gpt-4o")`,
`google("gemini-2.0-flash")` — so no registry is needed. Prefer a bare `"provider/model"` string
(for config-driven routing)? Pass a `providers:` registry at run time and the string resolves over
it. A string with no registry throws `NO_PROVIDER` — an [error that tells you the
fix](/concepts/errors/).

## Instructions: string or function

`instructions` is a `string`, or a function of the `RunContext` (highlighted) — sync or async — for
per-run system prompts built from `ctx.deps`. It's computed fresh each run and never captured in the
serialized state:

<Runnable title="dynamic-instructions.ts" highlight={[15]} code={`import { agent, createHarness } from "mithril";
import { scriptedProvider, testModel, textTurn } from "@mithril/core/testkit";

interface Deps { company: string }
const { agent: buildAgent } = createHarness<Deps>();

const model = testModel(scriptedProvider([textTurn("Your order shipped yesterday.")]));
const support = buildAgent({
  model,
  instructions: (ctx) => \`You help \${ctx.deps.company} customers. Be terse.\`,
});

await run(support, "Where is my order?", { deps: { company: "Acme" } });`} />

## Run it, or stream it

`run`, `stream`, and `resume` are different shapes, not aliases:

| Method | Returns | Use it when |
| --- | --- | --- |
| `agent.run(input)` | `Promise<RunResult>` | You want the final answer. |
| `agent.stream(input)` | `RunHandle` | You want to render tokens as they arrive. |
| `agent.resume(token, decision)` | `Promise<RunResult>` | You're resuming a [suspended](/guides/human-in-the-loop/) run. |

`RunResult` is a discriminated union — branch on `.status` (`"completed"`, `"suspended"`,
`"unresumable"`, `"error"`, `"cancelled"`; highlighted); never a bare try/catch. With no
dependencies, the options object is fully optional — the API key comes from `ANTHROPIC_API_KEY` /
`OPENAI_API_KEY`:

<Runnable title="run.ts" highlight={["12-14"]} code={`import { agent } from "mithril";
import { scriptedProvider, testModel, textTurn } from "@mithril/core/testkit";

const model = testModel(scriptedProvider([textTurn("It's 21°C and clear in Istanbul.")]));
const assistant = agent({ model, instructions: "You are a concise weather assistant." });

const result = await run(assistant, "What's the weather in Istanbul?");
switch (result.status) {
  case "completed":
    console.log(result.output);
    break;
  case "error":
    console.error(result.error.message);
    break;
  default:
    console.log(result.status);
}`} />

## Continuing a conversation

The `string` form is shorthand for a single `user` message. To send **chat history**, pass an array
of `InputMessage`s (`{ role: "user" | "assistant"; content: string }`, highlighted). There's no
`system` role — the system prompt comes only from `instructions`. You own the transcript: append the
model's answer and pass the grown array to the next call. Mithril holds no hidden session — that's
the [state = `replay(log)`](/concepts/state-is-replay/) principle:

<Runnable title="chat-history.ts" highlight={["6-10"]} code={`import { agent } from "mithril";
import { scriptedProvider, testModel, textTurn } from "@mithril/core/testkit";

const model = testModel(scriptedProvider([textTurn("About 2.1 million people.")]));
const assistant = agent({ model, instructions: "Answer follow-ups using the conversation so far." });

const history = [
  { role: "user", content: "What's the capital of France?" },
  { role: "assistant", content: "Paris." },
  { role: "user", content: "And its population?" },
] as const;

await run(assistant, history);`} />

<Aside type="note">
Threading the history yourself is deliberate — it keeps context construction visible and in your
control, with no framework guessing what to include. Durable *resume* of a single suspended run is a
separate, explicit [Checkpointer](/guides/memory-and-checkpointing/) concern.
</Aside>

<Aside type="tip" title="Outgrew the loop?">
`agent()` is the batteries-included loop. When you need full control, [eject
it](/guides/ejecting-the-loop/) — `agentLoop` is ~100 readable lines emitting the identical stream.
</Aside>
