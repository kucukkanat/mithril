`agent()` is the batteries-included loop. When you need full control over the drive — custom
scheduling, embedding the loop in your own runtime, or just reading exactly what happens — eject to
`agentLoop`. It is the same code `agent()` runs, roughly 100 readable lines, and it emits a
**byte-identical** `MithrilEvent` stream.

## Drive it yourself

`agentLoop` is an async generator: it *yields* `MithrilEvent`s and *returns* the final `RunResult`.
Because the result is the generator's return value, drive it to completion to read it (highlighted).
Run it — this is the same loop `agent()` runs:

<Runnable title="eject.ts" highlight={["16-22"]} code={`import { agentLoop, tool } from "mithril"; // or "@mithril/core/agent"
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

const weather = tool({
  name: "weather",
  description: "Current weather for a city.",
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ city }) => ({ city, tempC: 21 }),
});

const model = testModel(scriptedProvider([toolCallTurn("weather", { city: "Istanbul" }), textTurn("It's 21°C in Istanbul.")]));
const gen = agentLoop({ model, instructions: "Be concise.", tools: [weather], input: "Weather in Istanbul?", deps: undefined });

let result;
for (;;) {
  const step = await gen.next();
  if (step.done) { result = step.value; break; } // the RunResult
  if (step.value.type === "text.delta") console.log(step.value.delta);
}
console.log("status:", result.status);`} />

Every event here is identical to what `agent({ … }).stream(…).events` would produce — same `seq`,
same `span`, same order. That is the contract: [the protocol is the product](/concepts/the-protocol/),
so swapping the producer changes nothing downstream.

## Extensions carry over

The same middleware you compose on `agent()` work here — you just pass them directly. At the
`agent()` layer, `use:` is sugar that flattens plugins and middleware into two arrays; at the loop
layer you pass those arrays yourself (highlighted):

<Runnable title="eject-middleware.ts" highlight={["12-13"]} code={`import { agentLoop } from "mithril";
import { scriptedProvider, testModel, textTurn } from "@mithril/core/testkit";

const logModel = { name: "log-model", model: async (ctx, call, next) => { console.log("model call"); return next(call); } };
const seen = [];

const gen = agentLoop({
  model: testModel(scriptedProvider([textTurn("Hello there.")])),
  instructions: "Be concise.",
  input: "Hi",
  deps: undefined,
  middlewares: [logModel], // same Middleware objects as use:
  consumers: [{ name: "log", onEvent: (e) => seen.push(e.type) }],
});

for (;;) { const s = await gen.next(); if (s.done) break; }
console.log("consumer saw", seen.length, "events");`} />

`LoopOptions` also accepts `transport`, `providers`, `runtime`, `signal`, `maxSteps`, `output`,
`healing` (the pluggable self-healing stack — `false`/`[]` for a raw loop, or an array of healing
middleware), and — for continuing a suspended run — `runId` and `resume`.

<Aside type="tip" title="Eject last, not first">
Reach for `agentLoop` only when `run`/`stream` genuinely aren't enough. The point of the eject
hatch is that it exists and is small — not that you should live in it. Most apps never do.
</Aside>
