A tool is four things: a `name`, a `description` the model reads, an `inputSchema`, and an
`execute` function. Nothing else is required — input and output types are inferred, so you never
restate a generic:

```ts
import { tool } from "mithril";
import { z } from "zod"; // any Standard Schema validator

const weather = tool({
  name: "weather",
  description: "Current weather for a city.",
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ city }) => ({ city, tempC: 21 }), // ({ city }) is typed; returns { city, tempC }
});
```

That's the whole tool. Below is the same definition wired into an agent that runs **offline in your
browser** — click **Open in playground** (the scripted `model` and `run`/`toolCallTurn` helpers are
playground scaffolding; in your app you pass a real `anthropic(…)` handle and call `agent.run(...)`):

<Runnable title="weather-tool.ts" highlight={["5-10"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

const weather = tool({
  name: "weather",
  description: "Current weather for a city.",
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ city }) => ({ city, tempC: 21 }),
});

const model = testModel(scriptedProvider([toolCallTurn("weather", { city: "Istanbul" }), textTurn("It's 21°C in Istanbul.")]));
await run(agent({ model, instructions: "Use weather, then answer.", tools: [weather] }), "Weather in Istanbul?");`} />

## Any Standard Schema validator

`inputSchema` accepts anything implementing [Standard Schema](https://standardschema.dev) — Zod,
Valibot, and ArkType all work unchanged, because the tool only touches the `~standard` interface.
The `execute` argument is typed as the schema's **output**, so validation and types come from one
source.

## Inferred I/O, plus an output schema

`In` is recovered from the schema; `Out` from `execute`'s naked return value — so a tool that
returns an object does not collapse to `string`. `Out` must be a `JsonValue`. Add an `outputSchema`
(highlighted) to validate what you return, too:

<Runnable title="price-tool.ts" highlight={[8]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

const price = tool({
  name: "price",
  description: "Latest price for a ticker.",
  inputSchema: z.object({ ticker: z.string() }),
  outputSchema: z.object({ ticker: z.string(), usd: z.number() }),
  execute: async ({ ticker }) => ({ ticker, usd: 187.4 }),
});

const model = testModel(scriptedProvider([toolCallTurn("price", { ticker: "AAPL" }), textTurn("AAPL is $187.40.")]));
await run(agent({ model, instructions: "Look up the price, then answer.", tools: [price] }), "Price of AAPL?");`} />

## Typed dependencies

`execute`'s second argument is the `RunContext`. Bind your dependencies once with
[`createHarness`](/concepts/three-verbs/) and `ctx.deps` is typed in every tool (highlighted) — no
per-call `<Deps>()`. `deps` are re-injected every run and never serialized:

<Runnable title="typed-deps.ts" highlight={[16]} code={`import { agent, createHarness } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

interface Deps { users: Map<string, { name: string }> }
const { agent: buildAgent, tool } = createHarness<Deps>();

const getUser = tool({
  name: "get_user",
  description: "Look up a user by id.",
  inputSchema: z.object({ id: z.string() }),
  execute: async ({ id }, ctx) => ctx.deps.users.get(id) ?? { name: "unknown" }, // ctx.deps: Deps
});

const model = testModel(scriptedProvider([toolCallTurn("get_user", { id: "42" }), textTurn("That's Ada.")]));
const app = buildAgent({ model, instructions: "Look up the user, then answer.", tools: [getUser] });
await run(app, "Who is user 42?", { deps: { users: new Map([["42", { name: "Ada" }]]) } });`} />

<Aside type="note">
`ctx` also carries `runId`, `step`, `signal`, `usage`, `runtime`, `emit`, and `journal`. In the
playground, `run(agent, input, opts)` forwards `opts` (like `deps`) to the underlying run.
</Aside>

## Approval gates and progress

Set `needsApproval: true` (a boolean, or a predicate over the input) to pause the run before
`execute` — see [Human-in-the-loop](/guides/human-in-the-loop/). For long work, make `execute` an
async generator that yields `{ payload }` progress (highlighted); each yield emits a `tool.progress`
event:

<Runnable title="progress-tool.ts" highlight={["10-11"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { z } from "zod";

const index = tool({
  name: "index",
  description: "Index a document.",
  inputSchema: z.object({ url: z.string() }),
  execute: async function* ({ url }) {
    yield { payload: { phase: "fetch" } };
    yield { payload: { phase: "embed" } };
    return { indexed: true, url };
  },
});

const model = testModel(scriptedProvider([toolCallTurn("index", { url: "https://example.com" }), textTurn("Indexed.")]));
await run(agent({ model, instructions: "Index the URL, then confirm.", tools: [index] }), "Index example.com");`} />
