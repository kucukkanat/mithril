Evals in Mithril are **trajectory-native**: a run's recorded event log *is* the fixture, and a
scorer is a pure function over it. Nothing to mock — you assert against the same real `MithrilEvent`
stream the loop produced (and your app consumes).

## A complete example

A tool, an agent, one case, three scorers, and a run — copy-paste runnable with no key and no
network (the model is the scripted test double). The scorers (highlighted) assert the run completed,
called the right tool, and called it with the right arguments:

<Runnable title="weather-agent.ts" highlight={["24-26"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { runEval, calledTool, calledToolWith, completed } from "@mithril/evals";
import { z } from "zod";

const getWeather = tool({
  name: "getWeather",
  description: "Current weather for a city.",
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ city }) => ({ city, tempC: 21, sky: "clear" }),
});

// Swap this scripted model for openai("gpt-4o-mini") or a local transformers("…") — the cases don't change.
const model = testModel(scriptedProvider([toolCallTurn("getWeather", { city: "Istanbul" }), textTurn("It's 21°C and clear in Istanbul.")]));
const weatherAgent = agent({ model, instructions: "Use the weather tool to answer.", tools: [getWeather] });

const cases = [{
  name: "reports Istanbul weather",
  input: "What's the weather in Istanbul?",
  scorers: [
    completed(),
    calledTool("getWeather"),
    calledToolWith(getWeather, (i) => i.city === "Istanbul"), // typed input — no cast
  ],
}];

for await (const r of runEval(weatherAgent, cases)) {
  console.log(r.passed ? "✓" : "✗", r.case);
  for (const s of r.scores) console.log(\`   \${s.name} = \${s.value}\`);
}`} />

An eval is three parts: the **agent under test**, the **cases** (a named `input` + its `scorers`),
and the **scorers** (pure functions over the run). `runEval` streams one `EvalRun` — `{ case, scores,
trajectory, passed }` — per case; a case `passed` when every score meets the `threshold` (default `1`).

## Scorers

A `Scorer` is `(trajectory, ctx?) => Score`, where `Score` is `{ name, value, rationale? }`. Built-ins
cover tool-trajectory checks (`calledTool`, `calledToolWith`, `calledInOrder`, `toolCallCount`,
`noToolErrors`), output (`completed`, `outputIncludes`, `outputMatches`), and budget (`underCost`,
`underSteps`). A custom scorer reads the same `Trajectory` (highlighted) — `t.log` is the
`MithrilEvent[]`, `t.final` the folded `RunState`:

<Runnable title="custom-scorer.ts" highlight={["6-9"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { runEval } from "@mithril/evals";
import { z } from "zod";

// 1 only if the agent used exactly one tool call — no thrashing, no redundant calls.
const oneToolCall = (t) => {
  const calls = t.log.filter((e) => e.type === "tool.call").length;
  return { name: "one-tool-call", value: calls === 1 ? 1 : 0, rationale: \`\${calls} call(s)\` };
};

const search = tool({ name: "search", description: "Search.", inputSchema: z.object({ q: z.string() }), execute: async ({ q }) => ({ hits: 3, q }) });
const model = testModel(scriptedProvider([toolCallTurn("search", { q: "mithril" }), textTurn("Found 3.")]));
const agentUnderTest = agent({ model, instructions: "Use search.", tools: [search] });

for await (const r of runEval(agentUnderTest, [{ name: "one call", input: "search mithril", scorers: [oneToolCall] }])) {
  console.log(r.case, "→", r.scores[0].value);
}`} />

An **LLM-judge** is nothing special — read the transcript from `t.log`, ask a model to grade it, return
the `value`. The built-in `llmJudge({ model, rubric })` does exactly that.

## Record once, replay from file

`runEval` streams the agent live every time. For **deterministic, network-free** scoring, record each
case's trajectory once and replay it (highlighted). The example uses the in-memory store; on a server,
swap `memoryTrajectoryStore()` for `fsTrajectoryStore(nodeFileSystem({ root: "./fixtures" }))`:

<Runnable title="record-replay.ts" highlight={["12-16"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { runEvalCached, memoryTrajectoryStore, calledTool } from "@mithril/evals";
import { z } from "zod";

const getWeather = tool({ name: "getWeather", description: "Weather.", inputSchema: z.object({ city: z.string() }), execute: async ({ city }) => ({ city, tempC: 21 }) });
const model = testModel(scriptedProvider([toolCallTurn("getWeather", { city: "Istanbul" }), textTurn("It's 21°C.")]));
const weatherAgent = agent({ model, instructions: "Use the weather tool.", tools: [getWeather] });
const cases = [{ name: "istanbul", input: "Weather in Istanbul?", scorers: [calledTool("getWeather")] }];

const store = memoryTrajectoryStore();
// Record once — the only time the model runs (do this against the real provider):
for await (const _ of runEvalCached(weatherAgent, cases, { mode: "record", store })) void _;
// Replay — reads the stored log, re-scores, never touches the network:
for await (const r of runEvalCached(weatherAgent, cases, { mode: "replay", store })) {
  console.log("replayed:", r.case, r.passed);
}`} />

Fixtures are keyed by case name **+ input hash**, so changing a case's input misses the stale
recording. `serializeTrajectory` / `loadTrajectory` are the codec if you roll your own store.

## A visual HTML report

`htmlReport` turns a batch of results into a single **self-contained HTML string** (highlighted) —
summary stats plus cases you can filter by pass/fail and group, and search. No deps, no network:

<Runnable title="report.ts" highlight={["11-12"]} code={`import { agent, tool } from "mithril";
import { scriptedProvider, testModel, toolCallTurn, textTurn } from "@mithril/core/testkit";
import { runEval, htmlReport, calledTool } from "@mithril/evals";
import { z } from "zod";

const getWeather = tool({ name: "getWeather", description: "Weather.", inputSchema: z.object({ city: z.string() }), execute: async ({ city }) => ({ city, tempC: 21 }) });
const model = testModel(scriptedProvider([toolCallTurn("getWeather", { city: "Istanbul" }), textTurn("It's 21°C.")]));
const weatherAgent = agent({ model, instructions: "Use the weather tool.", tools: [getWeather] });

const entries = [];
for await (const r of runEval(weatherAgent, [{ name: "istanbul", input: "Weather in Istanbul?", scorers: [calledTool("getWeather")] }])) {
  entries.push({ run: r, group: "scripted", durationMs: 0 });
}
const html = htmlReport(entries, { title: "Weather agent evals" }); // in your app: await Bun.write("report.html", html)
console.log("report bytes:", html.length);`} />

Set `group` to the model id and push entries from several models to compare them side by side. For a
whole suite with a CI pass-rate gate, use `runSuite(entries, { minPassRate })`.

<Aside type="tip" title="A real, worked example">
The repo's `apps/evals` workspace runs exactly this shape against real
[local models](/guides/local-inference/) via `runSuite`, ending in an `htmlReport`. Run it with
`bun run evals`. `describeEval(test, agent, cases)` bridges any of this to `bun:test` / Vitest — one
named test per case, so a failing eval fails the suite. Keep eval and unit runs on separate commands.
</Aside>
