Small models (0.5B–8B, including the in-browser ones) get tool calls and JSON *almost* right. Mithril's
job is to close that gap **without magic**: every correction is visible as a typed
[`MithrilEvent`](/reference/core/protocol/type-aliases/mithrilevent/), bounded by a budget, and switchable
with one flag. Nothing is inferred from the model's name, and nothing rewrites the prompt behind your back.

This page is the complete reference: the principle, every correction and the event it emits, how to read
them, and how to turn any of them off.

## The one rule

The single most replicated finding in the literature is that a model **cannot reliably judge its own
answer** — intrinsic "review your work" prompting *degrades* accuracy, and it degrades *most* on small
models. So Mithril never asks the model to grade itself. Every correction is driven by an **external
verdict**:

- a Standard-Schema validation failure on a tool call or the final output,
- a tool that threw or returned the wrong shape,
- a duplicate-call signature,
- a token / cost / step budget.

Mithril already owns the two best verdict sources there are — schema validation at every tool boundary, and
tool results flowing through the event stream — so it can correct **from facts, not from vibes**.

<Aside type="tip" title="The contract">
Read the mechanism, not the marketing. If the harness repairs, re-asks, steers, or halts, there is an
event that says exactly why — and one flag that turns it off. Open the [devtools](/guides/devtools/) and you
can watch the whole correction trajectory unfold.
</Aside>

## What runs by default

These are on for every agent, cost nothing on the happy path, and are all pluggable middleware you can drop
or reconfigure (see [Toggling](#toggling-corrections)). The whole stack is `healing.defaults()` — one named
[`Middleware`](/reference/core/protocol/interfaces/middleware/) per behavior — installed automatically unless
you override the `healing` field.

| Correction | The failure it fixes | What the harness does | Middleware | Event |
| --- | --- | --- | --- | --- |
| **Parse-repair** | Malformed JSON from a local model (code fences, trailing commas, unterminated strings/brackets) | Repairs the text before validation — losslessly for valid JSON | — (provider) | — (wire-level) |
| **Schema coercion** | Args emitted as a JSON *string* instead of an object | Decodes and re-validates the string, once | `healing.argRepair` | `tool.repair` |
| **Bounded re-ask** | A tool called wrong, repeatedly | Re-asks with the concrete error; after the retry budget (default 2) consecutive failures, ends with a clear terminal error instead of looping to `maxSteps` | `healing.retryBudget` | `tool.retry` |
| **Loop detection** | The same `(tool, args)` call over and over | Nudges the model once, then halts | `healing.loopGuard` | `loop.detected` |
| **Output re-ask** | Final structured output that fails its `output` schema | Re-asks with the validation error (default 2 times), then a typed `OutputInvalid` | `healing.outputRetry` | `object.invalid` |
| **Output validation** | A tool returning data that breaks its own `outputSchema` | Rejects it and feeds the violation back | — (loop) | `tool.error` (`invalid_output`) |
| **Crash-hardening** | A provider or middleware that throws mid-run | Converts it to a typed error — never an uncaught crash | — (loop) | `run.error` |

<Aside type="note">
Two of these are *not* healing middleware. **Parse-repair** is a wire-parsing concern that lives in the
local-model provider (like the existing code-fence stripping), so it is always on. **Crash-hardening** is a
safety invariant wired into the loop — it stays on even with `healing: false`. Everything else is a
composable [`Middleware`](/reference/core/protocol/interfaces/middleware/) in the `healing` stack.
</Aside>

## The correction events

Every correction is a first-class member of the [`MithrilEvent`](/reference/core/protocol/type-aliases/mithrilevent/)
union, so devtools, OTel, and your own consumers all see them with full static typing.

| Event | Fires when | Key fields |
| --- | --- | --- |
| `tool.repair` | A malformed call was deterministically fixed | `mechanism` (`"parse"` \| `"coerce"`), `before`, `after` |
| `tool.retry` | A failing call is being re-asked under budget | `attempt`, `errorClass` |
| `tool.error` | A tool call failed (unknown tool, bad args, bad output, handler threw) | `error` (carries a `ToolErrorClass`) |
| `loop.detected` | An identical-call loop was caught | `signature`, `count`, `action` (`"steer"` \| `"halt"`) |
| `object.invalid` | Structured output failed its schema and is being re-asked | `attempt`, `issues` |
| `budget.exceeded` | A token / cost budget was passed | `budget` (`"tokens"` \| `"cost"` \| …), `limit`, `actual` |
| `run.error` | A provider/middleware threw and was caught | `error` |

### The tool-error taxonomy

Every tool failure carries a machine-readable [`ToolErrorClass`](/reference/core/protocol/type-aliases/toolerrorclass/)
on its error `data`, so you can route your own handling or bucket metrics without parsing message text:

```ts
import { toolErrorClass } from "mithril";

for await (const event of run.events) {
  if (event.type === "tool.error") {
    switch (toolErrorClass(event.error)) {
      case "unknown_tool":  // model named a tool that doesn't exist
      case "malformed_json": // arguments couldn't be parsed
      case "invalid_args":   // parsed, but failed the input schema
      case "invalid_output": // the tool's result failed its output schema
      case "handler_error":  // the tool's execute() threw
      case "timeout":        // the tool exceeded its time budget
    }
  }
}
```

## Toggling corrections

The whole self-healing stack is **pluggable middleware**, controlled by a single `healing` field on
[`AgentConfig`](/reference/core/agent/interfaces/agentconfig/) (per-agent) and on `RunOptions` (per-run; the
per-run value wins). It takes three shapes:

- **omitted** ⇒ the batteries-included default ([`healing.defaults()`](/reference/core/agent/functions/defaults/) — arg-repair, loop guard, retry budget, output retry),
- **`false` or `[]`** ⇒ the **raw loop** — no coercion, no loop detection, unbounded tool retries, no output re-ask,
- **an explicit array** ⇒ exactly those healing middleware, each independently configurable.

<Tabs>
<TabItem label="Raw loop">

`healing: false` gives you the **raw loop** — the pre-correction behavior, in one flag.

```ts
const raw = agent({
  model,
  instructions,
  tools,
  healing: false, // just the loop — no coercion, no loop guard, unbounded retries, no output re-ask
});
```

<Aside type="caution" title="What stays on">
`healing: false` does **not** disable **crash-hardening**. A provider or middleware that throws still
becomes a typed `run.error` rather than taking the process down — that's a safety invariant wired into the
loop, not a healing middleware, so it is never switchable.
</Aside>

</TabItem>
<TabItem label="Pick & configure">

Pass an array to install exactly the behaviors you want, each with its own options. The `healing` namespace
exports one factory per behavior:

```ts
import { agent, healing } from "mithril";

const a = agent({
  model,
  instructions,
  tools,
  // a raw loop except: coerce string args, and halt identical-call loops sooner
  healing: [healing.argRepair(), healing.loopGuard({ steerAt: 2, haltAt: 3 })],
});
```

| Factory | Governs | Options (defaults) |
| --- | --- | --- |
| [`healing.argRepair()`](/reference/core/agent/functions/argrepair/) | Schema coercion (the `tool.repair` events) | — |
| [`healing.retryBudget()`](/reference/core/agent/functions/retrybudget/) | The per-tool re-ask budget | `max` (2); `max: 0` ⇒ give up on first failure |
| [`healing.loopGuard()`](/reference/core/agent/functions/loopguard/) | Identical-call loop detection | `steerAt` (3), `haltAt` (4) |
| [`healing.outputRetry()`](/reference/core/agent/functions/outputretry/) | Structured-output re-ask | `max` (2) |

</TabItem>
<TabItem label="Extend the defaults">

To *add* to the default stack rather than replace it, spread
[`healing.defaults()`](/reference/core/agent/functions/defaults/) and append your own middleware — healing is
just middleware, so your own guardrail composes identically:

```ts
import { agent, healing } from "mithril";

const a = agent({
  model,
  instructions,
  tools,
  healing: [...healing.defaults(), myCustomGuard()],
});
```

</TabItem>
</Tabs>

<Aside type="tip" title="Healing is just middleware">
Every healing behavior acts only through the public middleware seam — reading step/tool/finalize outcomes
and driving `ctx.steer()` / `ctx.halt()` / `ctx.emit()`. There is no privileged path: your own middleware in
`healing` (or `use`) can steer and halt exactly the same way. See the [middleware
guide](/guides/middleware/) for the altitudes and context API.
</Aside>

## Budgets

Every run is bounded. `maxSteps` (default 16) always applies; the token and cost budgets are opt-in (unset ⇒
unbounded). Exceeding any of them emits [`budget.exceeded`](/reference/core/protocol/type-aliases/mithrilevent/)
and settles into an `error` result — **bounded and loud, never a silent give-up**. These are loop-level
budgets (not healing middleware) and stay on regardless of `healing`.

```ts
const assistant = agent({
  model: "openai/gpt-4o-mini",
  instructions: "Be helpful and brief.",
  tools: [weather],
  maxSteps: 16,            // model turns (default)
  maxTokens: 100_000,      // input+output tokens for the whole run
  maxCostMicroUsd: 50_000, // cost in integer micro-USD ($0.05)
});
```

## Steer small models with examples

The strongest *prompt-side* lift for tiny models is a handful of concrete examples. Attach them to a tool
and Mithril surfaces them into the wire description for every provider — no per-provider wiring:

```ts
const weather = tool({
  name: "get_weather",
  description: "Look up the current weather for a city.",
  examples: [{ city: "NYC" }, { city: "Paris" }],
  inputSchema: z.object({ city: z.string() }),
  async execute({ city }) {
    return `It is 22°C in ${city}.`;
  },
});
```

## Spend compute instead of parameters (opt-in)

When you want a small model to *try harder*, add a test-time-compute middleware to `use`. Both are **off
unless you add them** — a default run never spends N× tokens, and every sample is streamed and visible.

<CardGrid>
<LinkCard title="selfConsistency" href="/reference/core/agent/functions/selfconsistency/" description="Sample N times, keep the majority answer — voting by canonical equality of the tool calls or text." />
<LinkCard title="bestOfN" href="/reference/core/agent/functions/bestofn/" description="Draw N candidates, keep the highest-scoring one. Prefer a deterministic scorer over model self-judgement." />
</CardGrid>

```ts
import { agent, selfConsistency, bestOfN } from "mithril";

// majority vote across 5 samples, stopping early once 3 agree
const consistent = agent({ model, instructions, tools, use: [selfConsistency({ n: 5, earlyStopAgreement: 3 })] });

// draw 4 candidates, keep the one that actually called a tool (a deterministic verifier)
const verified = agent({ model, instructions, use: [bestOfN({ n: 4, score: (r) => (r.calls.length > 0 ? 1 : 0) })] });
```

<Aside type="note" title="In the browser, and beyond">
Samples run sequentially on one device in the browser, so keep `n` small (≤5). Asking a *small* model to
judge itself is weak — prefer a deterministic scorer (does it parse? does a check tool pass?). Hard
grammar-constrained decoding for local models is on the [roadmap](/roadmap/): Transformers.js has no grammar
support today, so parse-repair plus the bounded re-ask above is the shipped local guarantee.
</Aside>
